

# Fix: Infinite View Recording Loop

## Problem

An infinite re-render loop is spamming the view recording endpoint. All ~30,000+ view rows in the database were created in the last 8 minutes by a single browser tab. The root cause is a React dependency cycle:

```text
useEffect fires recordView()
  --> POST /views succeeds
  --> onSuccess invalidates 'story-views' query
  --> refetch triggers re-render
  --> recordViewMutation gets new identity (useMutation is not referentially stable)
  --> recordView (useCallback) gets new identity
  --> useEffect sees recordView changed in its deps
  --> fires recordView() again
  --> infinite loop
```

## Fix (2 files)

### 1. `src/hooks/use-story-views.ts` -- Stabilize `recordView`

- Replace `useCallback` with a `useRef`-based approach so `recordView` has a **stable identity** across renders
- Add a `hasRecorded` ref that tracks whether a view was already recorded for the **current** `storyId`, resetting when the storyId changes
- Remove `queryClient.invalidateQueries` from `onSuccess` -- the view count will refresh on its own via `staleTime`, and this invalidation is half of the infinite loop trigger
- This ensures exactly **1 POST per story opened**, no matter how many re-renders happen

### 2. `src/components/app/stories/StoryViewerModal.tsx` -- Clean up the useEffect

- Remove `recordView` from the useEffect dependency array (it will be stable now, but belt-and-suspenders)
- The effect should only depend on `isOpen` and `currentStory?.id`

### 3. Database cleanup -- Purge inflated view rows

- Delete all rows from `story_views` to reset counts to 0
- The counts are entirely fake (generated by the loop), so nothing real is lost

## What stays the same

- Every time a user opens a story, 1 view is recorded (no auth needed)
- View count is fetched via the GET endpoint as before
- No multiplier (already removed)

## Expected result

- View counts start at 0 and increment by 1 per actual story open
- No more runaway network requests in the console
- Database stays clean
